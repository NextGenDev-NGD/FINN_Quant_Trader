FINN framing rule: Every trade starts as a hypothesis. A hypothesis is only as good as the test behind it. FINN's workflow exists so that the process from "I think X works" to "I am trading X" is never skipped, compressed, or guessed. No IC, no trade. No out-of-sample test, no capital allocation. Research
1
Define the Hypothesis
Python · Markdown
State the edge in one testable sentence: "When X condition is true, Y outcome occurs Z% of the time." If you can't write the null hypothesis, you don't have a hypothesis — you have an observation.
Name the signal Define universe Set lookback window Write null hypothesis
Output: Hypothesis statement + null in plain English
▼ FINN check + common mistake
FINN Check
Can you state the null in one sentence? Is the universe, lookback window, and outcome period all defined? Example null: "RSI < 30 does NOT predict positive 5-day returns with IC > 0.05 in S&P 500 stocks over 2020–2024." If you can't name the null, stop here.
Common Mistake
"RSI below 30 predicts a reversal" — no universe, no lookback period, no definition of reversal. This is an observation masquerading as a hypothesis. Specificity is not optional: it is what makes the hypothesis falsifiable.
2
Pull and Clean the Data
yfinance · pandas
Download OHLCV data for the full lookback period. Flatten columns, handle splits, fill or drop NaNs. Log your data source and date range. Clean data is a prerequisite — not an afterthought.
yf.download() Flatten columns Check for gaps Document date range
Output: Clean OHLCV DataFrame with verified DatetimeIndex
▼ FINN check + common mistake
FINN Check
After yf.download(), flatten the multi-level column index: df.columns = [c[0] if isinstance(c, tuple) else c for c in df.columns]. Verify DatetimeIndex is continuous with no weekend gaps or duplicate rows. Check for split-adjusted close vs. Adj Close — use Adj Close for returns calculation.
Common Mistake
Accessing df['Close'] before flattening the multi-level column — produces a KeyError or silently returns the wrong data. The yfinance multi-ticker download wraps all columns in a second level. Always flatten first, verify second.
◆ Gate 1 — Data complete, gap-free, and date range verified? Fix before any modeling. Pre-Model
Build
3
Engineer the Features
pandas_ta · pandas
Compute technical indicators and derived features that define your signal. Each feature must have a clear directional role — no kitchen-sink feature dumps. Document why each feature is hypothesized to predict the target.
pandas_ta indicators Rolling windows Lag variables (.shift) Target variable
Output: Feature DataFrame — each column named and documented
▼ FINN check + common mistake
FINN Check
Does each feature exist at signal time without look-ahead? Use .shift(1) for any feature computed from data that closes on the same bar you're signaling from. The signal fires at close of day T — it cannot use T's close in its calculation.
Common Mistake
Computing today's RSI and using it as a signal for today's close — the close is one of the inputs to today's RSI. This is look-ahead bias. Build all features from prior-bar data. When in doubt, shift everything by 1.
4
Run the In-Sample Backtest
xgboost · scikit-learn
Train on the in-sample period (first 70–80% of data). Generate signals. Compute IC, win rate, Sharpe ratio, and max drawdown. This is your baseline performance — not your truth. Do not over-optimize here.
Time-based train/test split IC calculation Signal generation Equity curve
Output: IS performance — IC, Sharpe, drawdown, win rate
▼ FINN check + common mistake
FINN Check
Split by time — first 70–80% of dates for training, remainder for OOS. Did you shuffle the data at any point? If yes, invalidate all results and restart. Never use random train_test_split on time-series data.
Common Mistake
Using sklearn's random train_test_split on time-series. Shuffling breaks temporal order and inflates IS IC by allowing future returns to be correlated with past signal values. The result looks great. It means nothing.
◆ Gate 2 — IS IC > 0.05? If not, revise the hypothesis or features. Do not proceed to OOS. IC > 0.05
Test
5
Out-of-Sample Validation
scikit-learn · pandas
Evaluate the IS-trained model on the held-out period (last 20–30% of data). No retraining allowed. If IC collapses OOS, the in-sample edge was overfitting — not a real edge. This test is the most important one.
Apply IS model to OOS Compute OOS IC Compare IS vs. OOS Sharpe Log IC decay ratio
Output: OOS IC, Sharpe, drawdown — compared to IS baseline
▼ FINN check + common mistake
FINN Check
Apply the IS-trained model to OOS data with zero retraining. Record: OOS IC, OOS Sharpe, and the IS/OOS IC ratio. A ratio above 0.7 is healthy. Below 0.5 suggests overfitting. Below 0.3 — rebuild.
Common Mistake
Tuning any parameter using OOS performance as feedback — this converts OOS data into a second training set and invalidates the entire test. The OOS period is sacred. Once you look at it, it is no longer out-of-sample.
6
Stress Test the Edge
pandas · numpy
Test the strategy across different market regimes (trending, choppy, crisis). Vary key parameters ±20% and confirm the edge isn't parameter-sensitive fragility. Robustness is a prerequisite for capital allocation.
Regime slicing Parameter sensitivity Drawdown stress test Transaction cost impact
Output: Stress report — performance across regimes, parameter stability
▼ FINN check + common mistake
FINN Check
Does the strategy survive regime slicing? Test separately: 2020–2021 bull, 2022 bear, 2024 volatile. IC must be > 0.03 in at least 2 of 3 regimes. A strategy that "works" only in one regime isn't an edge — it's beta dressed up as alpha.
Common Mistake
Testing only on full-period data that includes a prolonged bull market. The full-period IC looks good because the strategy is long-biased in a bull regime. Slice by regime first. Full-period results come last.
◆ Gate 3 — OOS IC > 0.03 AND stable across ±20% parameter variation? If not, return to Step 3. IC > 0.03
7
Paper Trade the Signal
Webull · Signal log
Run the live signal in Webull paper trading for a minimum of 20 trades or 4 weeks, whichever is longer. Record every entry, exit, and reason in the signal log. Compare live IC to backtest IC.
Live signal generation Paper orders in Webull Daily log entry Compare to backtest
Output: signal_report_YYYYMMDD.csv — live vs. backtest IC
▼ FINN check + common mistake
FINN Check
Signal generation timing audit: are you firing signals using only data available at entry time? If the signal fires at market open using prior-day data, you cannot use same-day prices. Simulate exactly how the live trade would execute — no retroactive signal confirmation.
Common Mistake
Calculating the signal at 4 PM using the day's close, then entering "at the close." You cannot buy at the same price used in your signal calculation. This is look-ahead bias in execution disguise. Signal using prior-day data → enter at next-day open.
◆ Gate 4 — Live IC within 25% of backtest IC? Max DD acceptable? Proceed to capital allocation. Live IC ≥ 75% BT
Trade
8
Size and Risk Allocation
Risk rules · Position sizing
Apply FINN's 6 risk rules to determine position size, max exposure, and stop placement before the first live trade. Capital allocation is a function of IC, not conviction.
Max 2% portfolio risk/trade ATR-based stops Sector concentration limit Kelly sizing (if IC validated)
Output: Position size per trade, max drawdown threshold for shutdown
▼ FINN check + common mistake
FINN Check
Calculate before clicking: Position size = (Account × 0.02) ÷ (Entry − Stop). If the resulting share count requires more capital than the 2% rule allows at your stop distance — either widen the stop (using ATR) or skip the trade. There is no shortcut.
Common Mistake
Calculating position size after deciding "how much you want to make." Backwards math: starting from a profit target and working backward to position size systematically inflates risk — especially when conviction is high and the setup "looks certain."
9
Monitor, Log, and Iterate
all_trades.csv · monthly review
Log every live trade in all_trades_[initials].csv. Review IC monthly. If live IC decays below threshold for 2 consecutive months, stop trading and return to Step 5 to re-validate. The loop never ends.
Daily P&L log Monthly IC review Sharpe trending? Iteration trigger
Output: Monthly report — live IC, Sharpe, drawdown vs. target
▼ FINN check + common mistake
FINN Check
Plot rolling 30-day IC monthly. Is it trending down? Signal decay precedes P&L decay by weeks or months. The kill switch triggers when monthly IC < 0.03 for 2 consecutive months — don't wait for P&L to confirm what IC already told you.
Common Mistake
Monitoring P&L instead of IC. A strategy can produce positive P&L for months after the edge has decayed — riding existing positions, lucky regime, survivor bias in the sample. IC leads P&L. Watch the right metric.
